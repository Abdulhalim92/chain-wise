## ChainWise — план реализации (ARCHITECT)

**Версия:** 1.0  
**Дата:** 27.02.2026

Этот документ описывает целевую архитектуру и поэтапный план реализации платформы ChainWise.  
Он опирается на требования из `ChainWise-TZ.md` и описание из `ChainWise-Project-Description.md`.

---

## 1. Целевая архитектура

### 1.1. Тип проекта

- Микросервисная ERP-платформа для управления цепочкой поставок.
- Backend: набор Go-сервисов, каждый со своей БД PostgreSQL.
- Взаимодействие:
  - Межсервисное: gRPC.
  - События: RabbitMQ (exchange `order_events`).
  - Внешний API: REST/JSON через единый API Gateway.
  - Real-time: WebSocket/SSE через Notification Service.

### 1.2. Основные сервисы и роли

- `iam-service`: аутентификация, авторизация, роли и права (в т.ч. право «создание заказа» на точке).
- `order-service`: жизненный цикл заказа, State Machine, QR-токены, публикация событий в RabbitMQ.
- `inventory-service`: учёт остатков, асинхронный резерв по событиям, реакции на ошибки резерва.
- `accounting-service`: финансовый учёт, закупки, проводки склад↔филиал.
- `delivery-service`: назначение заказов на курьеров (pull/push), подтверждение забора и доставки.
- `notification-service`: подписка на события заказов, рассылка обновлений по WebSocket/SSE.
- `api-gateway`: единая точка входа для клиентов (планшеты, экран склада).

Каждый сервис — отдельный Go-модуль (или подпакет в монорепозитории) с собственным контрактом `.proto`.

### 1.3. Пакетная структура (внутри каждого сервиса)

Рекомендуемая структура сервиса (пример для `order-service`):

- `cmd/order-service/` — `main` (запуск HTTP/gRPC, конфигурация).
- `internal/config` — загрузка конфигурации (env, файлы).
- `internal/http` — REST-обёртки поверх gRPC (если нужны) / интеграция с API Gateway.
- `internal/grpc` — реализация gRPC-серверов (handlers).
- `internal/app` — прикладная логика, use-cases (application layer).
- `internal/domain` — доменная модель и интерфейсы репозиториев.
- `internal/repository` — доступ к БД PostgreSQL.
- `internal/mq` — работа с RabbitMQ (publish/consume).
- `pkg/contracts` или внешний модуль — сгенерированные gRPC-стабы из `.proto`.

Для остальных сервисов структура аналогична, с учётом специфики (есть/нет MQ, БД).

### 1.4. Направление зависимостей

- Внутри сервиса:
  - `cmd` → `internal/*` (но не наоборот).
  - `internal/http`, `internal/grpc`, `internal/mq` → `internal/app`.
  - `internal/app` → `internal/domain` (интерфейсы репозиториев, сущности).
  - `internal/repository` → `internal/domain` (реализация интерфейсов).
  - `internal/domain` не зависит от инфраструктуры (БД, MQ, HTTP).
- Между сервисами:
  - Сервисы зависят **на контракты** (`proto`/`contracts`), а не друг от друга напрямую.
  - Взаимодействие по gRPC и через RabbitMQ, без общих БД.

### 1.5. Публичные контракты между слоями

- **Межсервисные контракты**:
  - `.proto`-файлы в отдельном репозитории/модуле `chainwise-contracts`.
  - Пакеты: `auth.proto`, `orders.proto`, `inventory.proto`, `accounting.proto`, `delivery.proto`, `notifications.proto`.
- **Внутрисервисные контракты**:
  - Интерфейсы репозиториев и MQ-клиентов в `internal/domain` / `internal/app`.
  - HTTP/gRPC-обёртки не «видят» внутреннюю БД напрямую, работают через application layer.

## 2. План реализации по шагам

Ниже — последовательность шагов (STEP #N).  
Каждый шаг описан: цель, затронутые файлы/пакеты, ожидаемый результат, риск.

### STEP #1. Инициализация репозитория и модулей

- **Цель:** подготовить общую структуру Go-проекта и базовый layout для сервисов.
- **Пакеты/файлы:**
  - Корень репо: `go.work` или монорепозиторий с несколькими модулями.
  - Каталоги: `services/iam-service`, `services/order-service`, `services/inventory-service`, `services/accounting-service`, `services/delivery-service`, `services/notification-service`, `gateway/`.
- **Результат:**
  - Единый Git-репозиторий с базовой структурой каталогов.
  - Инициализированный `go mod init` для каждого сервиса (или один модуль + подкаталоги).
- **Риск:** **low** (организационный шаг, без бизнес-логики).

### STEP #2. Репозиторий контрактов (`chainwise-contracts`)

- **Цель:** создать отдельный модуль для `.proto` и сгенерированного Go-кода.
- **Пакеты/файлы:**
  - Отдельный репозиторий/модуль `chainwise-contracts` (или папка `contracts/` в текущем).
  - Файлы: `auth.proto`, `orders.proto`, `inventory.proto`, `accounting.proto`, `delivery.proto`, `notifications.proto`.
- **Результат:**
  - Определены gRPC-сервисы и сообщения:
    - `AuthService` (login, validate token).
    - `OrderService` (CreateOrder, GetOrder, ListOrders, ValidateQR, TransitionStatus).
    - `InventoryService` (IncreaseStock, GetStock, опционально CancelReservation).
    - `AccountingService` (RegisterPurchase, GetPurchase).
    - `DeliveryService` (ClaimOrder, AssignOrder, ConfirmPickup, ConfirmDelivery).
    - `Notification` (при необходимости — meta-сервис или только события MQ).
  - Настроен `buf`/`protoc` и Makefile/скрипты генерации Go-кода.
- **Риск:** **medium** (важно аккуратно продумать контракты, чтобы минимизировать будущие изменения).

### STEP #3. Проектирование схем БД для всех сервисов

- **Цель:** спроектировать и задокументировать схемы БД для каждого сервиса, имеющего постоянное хранилище, до начала реализации миграций и репозиториев.
- **Пакеты/файлы:**
  - Документация (например, `docs/schemas/` или описание в репозитории): по одному разделу/файлу на сервис.
  - Опционально: черновики миграций или ER-диаграммы.
- **Результат:**
  - **Order Service:** таблицы заказов (id, branch_id, status, created_at, …), позиций заказа (order_id, product_id, quantity, …), связь с филиалами/курьерами; при необходимости таблица outbox (id, event_type, payload, created_at, processed_at).
  - **Inventory Service:** таблицы локаций (id, type — ЦС/филиал, …), остатков (location_id, product_id, quantity, reserved, …), резервов (order_id, product_id, quantity, …), операций списания/прихода.
  - **Accounting Service:** таблицы закупок (id, supplier_id, document_ref, total, created_at, …), позиций закупки, проводок склад↔филиал (order_id, from_location, to_location, amount, …), справочники поставщиков и аналитические срезы.
  - **IAM Service:** таблицы пользователей (id, login, password_hash, role, branch_id при необходимости), разрешений (роль/пользователь — право «создание заказа» и др.), сессий при использовании Redis — только ключи и TTL.
  - **Delivery Service:** таблицы назначений заказ↔курьер (order_id, courier_id, assigned_at, status), при необходимости кэш «витрины» доступных заказов.
  - **Notification Service:** постоянная БД не обязательна; при необходимости — подписки (client_id, role, branch_id, courier_id) для маршрутизации WebSocket.
  - Все схемы согласованы с доменной моделью и контрактами (STEP #2); документ доступен команде для реализации миграций в STEP #8, #10, #11, #5, #13.
- **Риск:** **medium** (неверная схема потребует правок миграций и кода).

### STEP #4. Базовая инфраструктура: конфигурация, логирование, health-check, HTTP middleware и gRPC interceptors

- **Цель:** унифицировать базовые инфраструктурные компоненты для всех сервисов, включая кросс-срезные цепочки для HTTP и gRPC (request_id, recovery, logging, при необходимости метрики и auth).
- **Пакеты/файлы:**
  - Общий пакет `internal/platform` или отдельный модуль `chainwise-platform`:
    - `config` (чтение env / файлов).
    - `logger` (обёртка над логгером).
    - `health` (HTTP/gRPC health-check).
    - **HTTP middleware:** `middleware` — цепочка: генерация/проброс `request_id`, recovery (panic → 500 + лог), логирование запроса/ответа (метод, путь, статус, длительность). Опционально: метрики, ограничение частоты запросов.
    - **gRPC interceptors (unary и stream):** аналог того же набора — внедрение/проброс `request_id` в metadata, recovery (panic → gRPC error + лог), логирование вызова (method, status, duration). Опционально: метрики; auth (проверка JWT в metadata) выносится в отдельный interceptor или в Gateway.
  - Во всех сервисах: подключение единых middleware к HTTP-серверу и единых interceptors к gRPC-серверу.
- **Результат:**
  - Все сервисы используют единый способ загрузки конфигурации и логирования.
  - Есть стандартный endpoint/метод для проверки здоровья сервиса.
  - Каждый HTTP-запрос и gRPC-вызов проходят через request_id, recovery и логирование; при панике процесс не падает, ошибка логируется и возвращается клиенту.
- **Риск:** **low** (технический шаг, но влияет на наблюдаемость и устойчивость).

**Требования к конфигурации и логированию (реализованы в STEP #4):**

- **Конфигурация (Viper):** без дефолтных значений; загрузка только из конфигурационного файла. Порядок поиска: сначала файл `.env` (если есть — читать из него), иначе файл `config.yml` (или путь из переменной `CONFIG_FILE`). Если ни один файл не найден — `Load()` возвращает ошибку (например `config.ErrNoConfigFile`). Сигнатура: `Load() (Config, error)`. Переменные окружения (PORT, GRPC_PORT, ENV, LOG_LEVEL, LOG_FORMAT, LOG_ADD_SOURCE) имеют приоритет над значениями из файла (после чтения файла). Обязательные поля: port, grpc_port, env, log_level, log_format; при отсутствии любого из них — возвращать явную ошибку. Обоснование: явная конфигурация снижает риск запуска с неожиданными дефолтами; приоритет .env даёт удобство локальной разработки; возврат ошибки обеспечивает fail-fast при отсутствии конфига.
- **Логирование (slog + опции):** уровень (debug, info, warn, error), формат вывода (json — для prod/аггрегаторов, text — для локальной разработки), опция добавления источника (файл:строка) для отладки, имя сервиса в каждом сообщении; поддержка дочерних логгеров (WithService, With) для обогащения контекста. Обоснование: структурированные логи и единый формат (slog) упрощают парсинг и мониторинг; возможность переключения формата и уровня без смены кода снижает операционные риски и ускоряет расследование инцидентов.

### STEP #5. Каркас IAM Service (миграции и репозитории по схеме из STEP #3)

- **Цель:** реализовать минимальный IAM: регистрацию/логин, выдачу JWT, проверку прав на основе ролей и разрешений.
- **Пакеты/файлы:**
  - `services/iam-service/cmd/iam-service/main.go`.
  - `services/iam-service/internal/{config,http,grpc,app,domain,repository}`.
- **Результат:**
  - gRPC/REST endpoint’ы для:
    - Аутентификации (получение JWT).
    - Валидации токена (используется Gateway и сервисами).
  - Модель ролей и разрешений (в т.ч. право «создание заказа» для оператора точки).
- **Риск:** **medium** (критично для безопасности, но относительно стандартная задача).

### STEP #6. Интеграция API Gateway с IAM

- **Цель:** обеспечить аутентификацию и авторизацию запросов через API Gateway.
- **Пакеты/файлы:**
  - `gateway/` (конфиг маршрутизации, middleware аутентификации).
- **Результат:**
  - Все внешние запросы к backend проходят через Gateway.
  - JWT проверяется, роли и разрешения извлекаются и прокидываются в бэкенды (в context/metadata).
- **Риск:** **medium** (ошибки здесь влияют на безопасность и UX).

### STEP #7. Каркас Order Service (без RabbitMQ)

- **Цель:** реализовать базовый CRUD и State Machine без событий.
- **Пакеты/файлы:**
  - `services/order-service/cmd/order-service/main.go`.
  - `services/order-service/internal/{config,grpc,http,app,domain,repository}`.
- **Результат:**
  - gRPC/REST методы:
    - `CreateOrder` (создаёт заказ в статусе Created, без реального резерва).
    - `GetOrder`, `ListOrders` (фильтрация по роли: филиал/курьер/склад).
  - Простая реализация State Machine в доменном слое (пока без всех переходов).
- **Риск:** **medium** (базовый функционал, но важно сразу заложить правильную модель).

### STEP #8. Интеграция Order Service с PostgreSQL (миграции и репозитории по схеме из STEP #3)

- **Цель:** реализовать миграции и репозитории для заказов по спроектированной схеме БД (STEP #3) и подключить PostgreSQL.
- **Пакеты/файлы:**
  - `services/order-service/internal/repository` (PostgreSQL-репозитории).
  - Миграции (например, `migrations/order-service`).
- **Результат:**
  - Таблицы заказов, позиций, при необходимости outbox — реализованы по документированной схеме.
  - Хранение заказов в PostgreSQL с учётом требований к State Machine и событиям.
  - Миграции запускаются при деплое/старте сервиса.
- **Риск:** **medium** (отклонения от схемы STEP #3 усложняют поддержку).

### STEP #9. Интеграция Order Service с RabbitMQ (publish событий)

- **Цель:** реализовать публикацию событий заказа в RabbitMQ при смене статуса.
- **Пакеты/файлы:**
  - `services/order-service/internal/mq` (клиент RabbitMQ).
- **Результат:**
  - Публикация событий в exchange `order_events` с routing key по статусу (`order.created`, `order.reserved`, ...).
  - Publisher Confirms и базовые retry при недоступности брокера.
- **Риск:** **medium** (нужно аккуратно обработать ошибки, чтобы не терять события).

### STEP #10. Каркас Inventory Service + подписка на `order.created` (миграции и репозитории по схеме из STEP #3)

- **Цель:** реализовать сервис остатков с миграциями и репозиториями по схеме из STEP #3 и асинхронным резервом при создании заказа.
- **Пакеты/файлы:**
  - `services/inventory-service/cmd/inventory-service/main.go`.
  - `services/inventory-service/internal/{config,grpc,app,domain,repository,mq}`.
- **Результат:**
  - Миграции и репозитории по документированной схеме остатков (STEP #3).
  - Подписка очереди `inventory_reserve` на `order.created`.
  - Асинхронный резерв остатков (схема retry при временных ошибках).
  - При исчерпании retry — вызов Order для перевода заказа в Cancelled / публикация `order.cancelled`.
- **Риск:** **high** (важен правильный дизайн идемпотентности и обработка ошибок).

### STEP #11. Каркас Accounting Service + интеграция с Inventory (миграции и репозитории по схеме из STEP #3)

- **Цель:** реализовать учёт закупок и проводок с миграциями и репозиториями по схеме из STEP #3 и интеграцией с Inventory.
- **Пакеты/файлы:**
  - `services/accounting-service/...`.
  - gRPC-вызовы `Inventory.IncreaseStock`.
- **Результат:**
  - Миграции и репозитории по документированной схеме учёта (STEP #3).
  - Endpoint для регистрации внешней закупки (кладовщик).
  - gRPC-вызов в Inventory для увеличения остатков на ЦС.
- **Риск:** **medium** (финансовая логика, но пока без сложной аналитики).

### STEP #12. Accounting Service как consumer `order.delivered`

- **Цель:** закрывать проводки по внутренним поставкам на основе событий доставки.
- **Пакеты/файлы:**
  - `services/accounting-service/internal/mq` (consumer очереди `accounting_events`).
- **Результат:**
  - Подписка на `order.delivered`.
  - Создание проводки «Склад → Филиал» при получении события.
- **Риск:** **medium** (важна идемпотентность — запрет двойных проводок).

### STEP #13. Каркас Delivery Service (назначение курьера; миграции и репозитории по схеме из STEP #3)

- **Цель:** реализовать оба сценария назначения курьера (pull/push).
- **Пакеты/файлы:**
  - `services/delivery-service/...`.
  - gRPC-вызовы к Order для чтения/обновления статуса (при необходимости).
- **Результат:**
  - Методы:
    - `ClaimOrder` (pull): курьер сам выбирает заказ из витрины.
    - `AssignOrder` (push): оператор филиала назначает заказ на конкретного курьера.
  - Сохранение привязки order_id ↔ courier_id.
- **Риск:** **medium** (нужно учесть конкуренцию при одновременном Claim нескольких курьеров).

### STEP #14. Каркас Delivery Service: подтверждение забора и доставки (QR/кнопка)

- **Цель:** реализовать полный цикл подтверждений курьера.
- **Пакеты/файлы:**
  - `services/delivery-service/internal/app` и интеграция с Order.
- **Результат:**
  - Методы:
    - `ConfirmPickup` (QR/кнопка) → `Order.ValidateQR` → `Order.TransitionStatus(InTransit)` → `order.in_transit`.
    - `ConfirmDelivery` (QR/кнопка) → `Order.ValidateQR` → `Order.TransitionStatus(Delivered)` → `order.delivered`.
- **Риск:** **high** (QR-токены, одноразовость, безопасность).

### STEP #15. Реализация QR-токенов в Order Service

- **Цель:** сделать безопасные QR-токены (ID + одноразовый токен).
- **Пакеты/файлы:**
  - `services/order-service/internal/domain` (модель токена).
  - `services/order-service/internal/app` (генерация, проверка, истечение).
- **Результат:**
  - QR содержит ID заказа и одноразовый токен.
  - `ValidateQR` проверяет токен, статус заказа и одноразовость.
- **Риск:** **high** (безопасность операций передачи груза).

### STEP #16. Каркас Notification Service + WebSocket

- **Цель:** обеспечить real-time обновление списков заказов для всех ролей.
- **Пакеты/файлы:**
  - `services/notification-service/...`.
  - Подписка на очередь `notification_events`.
- **Результат:**
  - WebSocket/SSE-ендпоинт, принимающий JWT и контекст (role, branch_id, courier_id).
  - Маршрутизация событий заказов к нужным клиентам:
    - Экран склада: все заказы.
    - Оператор точки: только по его филиалу.
    - Курьер: только его заказы.
- **Риск:** **medium** (масштабируемость, устойчивость подключений).

### STEP #17. API Gateway: маршрутизация ко всем сервисам

- **Цель:** завершить конфигурацию маршрутов для всех сервисов.
- **Пакеты/файлы:**
  - `gateway/config` (маршруты, правила аутентификации).
- **Результат:**
  - Внешние REST-ендпоинты доступны по единым путям (`/api/orders`, `/api/inventory`, `/api/accounting`, `/api/delivery`, `/api/auth`).
  - WebSocket для Notification проброшен через Gateway.
- **Риск:** **medium** (нужно следить за совместимостью контрактов и маршрутов).

### STEP #18. Обработка ролей и фильтрации в Order/Delivery

- **Цель:** выполнить продуктовые требования по видимости заказов.
- **Пакеты/файлы:**
  - `services/order-service/internal/app` (ListOrders с фильтрацией по роли/branch_id/courier_id).
  - `services/delivery-service/internal/app` (список «Мои заказы» курьера).
- **Результат:**
  - Оператор точки видит только свои заказы (по branch_id).
  - Кладовщик видит все заказы.
  - Курьер видит только назначенные ему заказы.
- **Риск:** **medium** (авторизация и фильтрация должны быть непротиворечивыми).

### STEP #19. Механизмы идемпотентности и retry

- **Цель:** защититься от повторной обработки событий и вызовов.
- **Пакеты/файлы:**
  - `services/order-service/internal/mq` (outbox-подход или хранение idempotency key).
  - `services/inventory-service/internal/mq` (идемпотентные обработчики `order.created`).
  - `services/accounting-service/internal/mq` (идемпотентная обработка `order.delivered`).
- **Результат:**
  - Повторная доставка одного и того же сообщения не приводит к двойному резерву или двойной проводке.
- **Риск:** **high** (критично для корректности данных).

### STEP #20. Технические нефункциональные требования

- **Цель:** реализовать общие нефункциональные требования.
- **Пакеты/файлы:**
  - Во всех сервисах: graceful shutdown, timeouts, метрики/логирование.
- **Результат:**
  - При SIGTERM/SIGINT сервисы корректно закрывают gRPC-соединения, HTTP-сервера и каналы RabbitMQ.
  - Базовые метрики (кол-во запросов, ошибок, задержки) и структурированные логи.
- **Риск:** **medium** (требует внимательности, но не меняет бизнес-логику).

### STEP #21. Интеграционные сценарии и end-to-end тесты

- **Цель:** проверить ключевые бизнес-потоки end-to-end.
- **Пакеты/файлы:**
  - Отдельный каталог тестов (например, `tests/e2e`).
- **Результат:**
  - Набор автоматизированных сценариев:
    - Внутренняя поставка (happy path).
    - Ошибка резерва и отмена.
    - Назначение курьера (pull/push).
    - Внешняя закупка и увеличение остатков.
  - Возможность поднять минимальное окружение (docker-compose) для запуска тестов.
- **Риск:** **medium** (объёмно, но даёт высокую уверенность).

## 2.1. Дополнительные архитектурные шаги

Ниже приведены дополнительные шаги, направленные на стандартизацию инфраструктуры, конфигурации, доставки событий и тестирования во всех сервисах ChainWise. Они снижают архитектурные риски и упрощают сопровождение.

### STEP #22. Единый подход к конфигурации (Viper + .env + config.yml)

- **Цель:** стандартизировать загрузку конфигурации во всех сервисах (единый механизм чтения config.yml, .env и переменных окружения).
- **Пакеты/файлы:**
  - Общий модуль (например, `chainwise-platform/config`) или `internal/platform/config`:
    - Поддержка `config.yml` и `.env` (через Viper и godotenv).
    - Единый `Config`-тип (Env, Port, DatabaseURL, RabbitMQURL и т.п.).
- **Результат:**
  - Одинаковый механизм конфигурации во всех сервисах.
  - Лёгкое переключение окружений (dev/stage/prod) и использование `.env`/`config.yml`.
- **Почему добавлено:** единый подход к конфигурации уменьшает количество ошибок, ускоряет онбординг и упрощает эксплуатацию.
- **Риск:** **low** (технический шаг, основанный на рабочем примере).

### STEP #23. Transactional Outbox для Order Service (и, при необходимости, других сервисов)

- **Цель:** формализовать и внедрить паттерн Transactional Outbox вместо абстрактного описания идемпотентности (STEP #19).
- **Пакеты/файлы:**
  - `services/order-service/internal/domain` и `internal/repository`:
    - Таблица `outbox` в схеме БД (миграции `migrations/order-service/002_add_outbox_*`).
    - Репозиторий, сохраняющий заказ и запись в outbox **в одной транзакции**.
  - `services/order-service/internal/relay`:
    - Компонент relay, читающий необработанные записи и публикующий события в RabbitMQ.
- **Результат:**
  - При создании/обновлении заказа события не отправляются в RabbitMQ напрямую, а записываются в outbox.
  - Фоновый relay гарантированно доставляет события в `order_events`, обеспечивая «at least once delivery».
- **Почему добавлено:** Transactional Outbox даёт гарантированную доставку событий (вместе с Publisher Confirms и DLX) и снижает риск потери сообщений между БД и брокером.
- **Риск:** **high** (критичный механизм, но опирается на проверенный дизайн).

### STEP #24. Стандартные handlers и адаптеры (HTTP/gRPC/consumer)

- **Цель:** закрепить структуру `internal/handlers` и `internal/infrastructure` для всех сервисов.
- **Пакеты/файлы:**
  - В каждом сервисе:
    - `internal/handlers/http` — HTTP-обработчики (REST API).
    - `internal/handlers/grpc` — gRPC-серверы (если используются).
    - `internal/handlers/consumer` — консьюмеры RabbitMQ/Kafka.
    - `internal/infrastructure/postgres`, `internal/infrastructure/rabbitmq`, `internal/infrastructure/payment` и т.п.
- **Результат:**
  - Единый подход: handlers только парсят входящие данные/формируют ответы и вызывают use-case (application layer).
  - Вся работа с БД, брокерами и внешними API сосредоточена в `internal/infrastructure`.
- **Почему добавлено:** единообразная структура слоёв улучшает предсказуемость, тестируемость и защищает зависимости (internal vs pkg).
- **Риск:** **medium** (меняет детализацию структуры, но не архитектурные принципы).

### STEP #25. Локальная среда разработки: Docker Compose, Makefile и миграции

- **Цель:** обеспечить быстрый запуск окружения и стандартные команды разработчика (Makefile, docker-compose, миграции).
- **Пакеты/файлы:**
  - В корне репозитория:
    - `docker-compose.yml` и `docker-compose.override.yml` с сервисами PostgreSQL, RabbitMQ, возможно Redis.
    - `Makefile` с командами (`make run`, `make migrate-up`, `make migrate-down`, `make lint`, `make test`).
  - Для каждого сервиса:
    - Каталог `migrations/<service-name>` с SQL-миграциями, интеграция с `golang-migrate` или аналогом.
- **Результат:**
  - Один `make up` или `docker-compose up` поднимает базовые зависимости (БД, MQ).
  - Стандартизированные команды сборки/запуска/миграций снижают барьер входа и упрощают CI/CD.
- **Почему добавлено:** общие инструменты для запуска и миграций сокращают время настройки окружения для новых разработчиков и автоматизируют операции.
- **Риск:** **medium** (операционный шаг, но требует внимательного сопровождения при изменении сервисов).

### STEP #26. Тестовые конвенции и вспомогательные пакеты

- **Цель:** ввести единый подход к тестам и test utilities (internal/testutil, mocks, builder/factory).
- **Пакеты/файлы:**
  - Общий пакет `internal/testutil` или модуль `chainwise-testutil`:
    - builders/factories для доменных сущностей.
    - helpers для случайных данных, фикстур и т.д.
  - По сервисам: table-driven tests для domain/usecase/repository.
- **Результат:**
  - Единый стиль тестов, возможность быстро писать новые кейсы.
  - Повышенная покрываемость доменного и application-слоя.
- **Почему добавлено:** единые тестовые конвенции уменьшают разброс стилей и упрощают code review.
- **Риск:** **low** (улучшение практик тестирования, без изменения бизнес-логики).

## 3. Итог

После выполнения STEP #1–#26 платформа ChainWise будет иметь:

- Полноценный backend из шести сервисов + API Gateway.
- Согласованные gRPC/REST-контракты и событийную модель через RabbitMQ.
- Проектирование схем БД для всех сервисов (STEP #3) и реализацию миграций/репозиториев по этим схемам.
- Единые HTTP middleware и gRPC interceptors (request_id, recovery, logging, опционально метрики и auth).
- Реализацию ключевых бизнес-процессов: внешняя закупка и внутренние поставки.
- Базовые механизмы безопасности, идемпотентности, логирования и graceful shutdown.
- Практически ориентированную инфраструктуру (конфигурация, Outbox, handlers/infrastructure, локальная среда разработки и тестовые конвенции).

План можно использовать как основу для режима CODER: в каждом запросе указывать конкретный STEP #N для реализации.
