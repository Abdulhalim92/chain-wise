# ChainWise

**ERP-платформа** для автоматизации цепочки поставок: внешняя закупка на центральный склад и внутренние поставки со склада в филиалы. Операции выполняются с планшетов; интерфейс зависит от роли; списки заказов обновляются в реальном времени по WebSocket.

## Возможности

| Область | Описание |
|--------|----------|
| **Внешняя закупка** | Учёт поступлений от поставщиков, накладные, цены, увеличение остатков на ЦС |
| **Внутренние поставки** | Заявки филиалов, резервирование, сборка, доставка курьером, подтверждение прихода по QR |
| **Роли** | Оператор точки (создание заказа, назначение курьера), кладовщик (сборка, QR, учёт закупок), курьер (забор/доставка по QR), администратор |

## Стек

| Компонент | Технология |
|-----------|------------|
| Backend | Go 1.26 |
| Межсервисное API | gRPC (Protocol Buffers v3) |
| Внешний API | REST/JSON через API Gateway |
| Очереди | RabbitMQ (topic `order_events`) |
| БД | PostgreSQL 16 (отдельная на сервис) |
| Сессии | Redis (IAM) |
| Real-time | WebSocket / SSE |

## Быстрый старт

### 1. Клонирование и переход в каталог

```bash
git clone https://github.com/Abdulhalim92/chain-wise.git
cd chain-wise
```

### 2. Генерация кода из контрактов

Требуется установленный **buf**:

```bash
cd contracts && make generate && cd ..
```

### 3. Сборка и запуск

**Gateway:**

```bash
go build -o bin/gateway ./gateway/cmd/gateway
./bin/gateway
```

**Сервис (например, IAM) из каталога сервиса:**

```bash
cd services/iam-service && go build -o ../../bin/iam-service ./cmd/iam-service && ../../bin/iam-service
```

> Перед запуском положите в рабочем каталоге `.env` или `config.yml` (см. [Конфигурация](#конфигурация-platform)).

## Конфигурация (platform)

Конфигурация загружается через **Viper** без дефолтов:

1. Сначала ищется файл **`.env`**
2. При отсутствии — **`config.yml`** (путь к YAML задаётся через `CONFIG_FILE`)
3. Если ни один файл не найден — сервис завершается с ошибкой

Переменные окружения **переопределяют** значения из файла.

| Переменная | Описание |
|------------|----------|
| `PORT`, `GRPC_PORT` | Порты HTTP и gRPC (допустимый диапазон 1–65535) |
| `ENV`, `LOG_LEVEL` | Окружение и уровень логов (`debug`, `info`, `warn`, `error`) |
| `LOG_FORMAT` | `json` (по умолчанию) или `text` |
| `LOG_ADD_SOURCE` | Добавлять в логи файл и строку вызова (`true`/`false`) |

Пример конфига: `platform/config/config.example.yml`.

**Рабочий каталог:** файлы `.env` и `config.yml` ищутся в каталоге запуска процесса. Для запуска из корня репо — положите конфиг в корень; из каталога сервиса — в каталог сервиса.

## Go workspace и зависимости

| Элемент | Назначение |
|--------|------------|
| **go.work** | Список модулей (contracts, platform, gateway, services/...) и версия Go |
| **go work use** | Добавить модуль: `go work use ./path/to/module` |
| **go work sync** | Обновить go.work.sum после изменений |
| **go mod tidy** | Выполнять в каталоге каждого модуля после изменения импортов |

Сборка из корня: `go build ./platform/... ./gateway/... ./services/iam-service/...`

## Логирование (platform)

- **slog** (Go 1.21+): структурированные логи, уровень, формат (JSON/текст), опционально источник (файл:строка), имя сервиса
- Дочерние логгеры: `WithService(name)`, `With(args...)`
- HTTP и gRPC проходят через middleware/interceptors с логированием метода, пути, статуса и длительности
